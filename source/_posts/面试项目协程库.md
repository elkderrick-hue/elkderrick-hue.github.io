# 手写协程库项目面试速记

这篇内容整理自我对一个 C++ 手写协程库项目的面试准备，目标不是逐行吃透源码，而是快速达到“能讲清项目、能回答高频问题、能稳住追问”的水平。

---

## 一、1 分钟项目自述

我做过一个基于 C++ 的轻量级用户态协程库，主要目标是解决高并发 IO 场景下线程阻塞和线程切换成本高的问题。  
项目整体包括 Fiber、Scheduler、TimerManager、IOManager 和 Hook 五个核心模块。Fiber 基于 ucontext 实现有栈协程，支持协程的创建、切换、挂起和恢复；Scheduler 负责统一调度协程和回调任务；IOManager 基于 epoll 管理 fd 事件；TimerManager 用于处理 sleep 和 IO 超时；Hook 模块则拦截 read、write、connect、sleep 等阻塞调用。  
当协程执行到暂时不可完成的 IO 操作时，Hook 不会阻塞线程，而是把当前协程挂起，并把对应事件注册到 epoll，等事件就绪后再恢复协程执行。这样业务层可以保持同步写法，但底层已经实现了协程化的异步调度。这个项目让我对协程调度、epoll、非阻塞 IO 和 Hook 机制有了比较系统的理解。

## 二、3 分钟展开版

这个项目本质上是一个轻量级协程运行时，核心目标是在 Linux 环境下，把传统会阻塞线程的 IO 操作改造成“挂起协程、不阻塞线程”的执行方式，从而提高 IO 密集型场景下的并发能力。

整体上我把它理解成五个模块。

第一个是 Fiber，也就是协程对象。项目基于 ucontext 实现了有栈协程，每个 Fiber 都有自己的栈空间、执行函数和上下文状态，通过 swapcontext 实现协程之间的切换。

第二个是 Scheduler，也就是调度器。它负责维护待执行任务队列，任务既可以是 Fiber，也可以是普通回调函数。调度器内部会启动多个工作线程，不断从队列里取任务执行；如果当前协程主动 yield，或者因为等待 IO 挂起了，调度器就去执行其他可运行任务。

第三个是 TimerManager，它负责处理时间相关的调度，比如 sleep、usleep 这种延时唤醒，以及 connect、read、write 这种 IO 操作的超时控制。

第四个是 IOManager，它是在调度器基础上结合 epoll 实现的。它负责监听 fd 的读写事件，并在事件到来时把对应协程重新加入调度队列。

第五个是 Hook 模块。这个模块比较关键，它通过 dlsym 拿到系统原始的阻塞函数，比如 read、write、connect、sleep 等，然后在协程环境下对这些函数做拦截。比如一个协程调用 read，如果当前 fd 没有数据，原本应该阻塞线程，但 Hook 会先把这个 fd 的读事件注册给 epoll，然后让当前协程 yield，线程去跑其他任务。等 epoll 检测到这个 fd 可读时，再把原来的协程恢复回来继续执行。

所以这个项目的核心价值在于：业务层写法还是同步风格，但底层通过协程调度、epoll 事件驱动和定时器机制，把阻塞线程改成了挂起协程，更适合高并发 IO 场景。

这个项目也让我更深入理解了几个问题，比如协程和线程的区别、epoll 在高并发场景下为什么高效、非阻塞 IO 和 EAGAIN 的处理方式、以及 Hook 如何把同步代码改造成异步执行流程。

## 三、15 个高频追问 + 标准回答

### 1. 你这个项目解决的核心问题是什么？

主要解决的是高并发 IO 场景下，线程容易被阻塞、线程数膨胀、上下文切换成本高的问题。这个项目通过“阻塞协程而不是阻塞线程”的方式，让一个线程可以复用执行多个 IO 任务，从而提高资源利用率。

### 2. 协程和线程的核心区别是什么？

线程是内核调度单位，由操作系统调度，切换需要进入内核态，成本相对更高；协程是用户态调度单位，由程序自己控制切换，切换更轻量。线程更适合利用多核并行，协程更适合 IO 密集型并发任务。

### 3. 为什么这个项目适合 IO 密集型，不适合 CPU 密集型？

因为协程的优势在于等待 IO 时可以主动让出执行权，让线程去做别的事。如果是 CPU 密集型任务，本质上计算一直占着 CPU，不会因为协程切换而减少总计算量，所以收益有限。CPU 密集型更依赖多线程或多进程利用多核。

### 4. Fiber 是怎么实现的？

项目基于 ucontext 实现有栈协程。每个 Fiber 保存自己的上下文、栈空间和执行函数，通过 getcontext、makecontext、swapcontext 来完成协程初始化和协程切换。切换时可以理解为保存当前执行现场，并恢复另一个协程之前保存的现场。

### 5. 为什么要用 thread_local？

因为协程调度是在线程内部完成的，每个线程都有自己的当前协程、主协程和调度协程状态。使用 thread_local 可以让这些状态在线程之间隔离，避免互相干扰。

### 6. 主协程、调度协程、任务协程分别是什么？

主协程一般是线程启动时对应的初始执行上下文；调度协程负责调度逻辑，比如从任务队列中取任务、切换到任务协程；任务协程则是真正承载业务逻辑的 Fiber。可以理解为主协程是基础上下文，调度协程是管理者，任务协程是执行者。

### 7. Scheduler 做了什么？

Scheduler 负责维护任务队列，并让多个工作线程循环取出任务执行。任务可以是 Fiber，也可以是普通函数回调。它还要处理空闲线程、活跃线程、任务唤醒等问题，是整个协程运行时的调度中心。

### 8. IOManager 和 Scheduler 是什么关系？

IOManager 可以理解为 Scheduler 的增强版。Scheduler 只负责普通任务调度，而 IOManager 在这个基础上增加了 epoll 事件管理能力，使协程不仅能被普通调度器调度，也能被 IO 事件唤醒。

### 9. Hook 的意义是什么？

Hook 的意义是保留同步编程接口，比如 read、write、connect、sleep 这些调用方式不变，但底层执行逻辑已经变成协程友好的异步流程。这样既避免了回调地狱，也让业务代码更直观。

### 10. read 被 hook 后，如果当前没数据，会发生什么？

Hook 会先调用原始 read，如果返回 EAGAIN，说明当前 fd 处于非阻塞模式且暂时没有数据可读。此时不会阻塞线程，而是把该 fd 的读事件注册到 epoll，随后让当前协程 yield。等 epoll 检测到该 fd 可读时，再把这个协程重新加入调度队列，恢复执行。

### 11. EAGAIN 在这里为什么很重要？

因为它不是致命错误，而是在非阻塞 IO 场景下告诉我们“现在还没准备好，稍后再试”。项目正是利用这个返回值，把“立即重试”改成“注册事件、挂起协程、等待唤醒”的流程。

### 12. TimerManager 在项目里起什么作用？

TimerManager 一方面负责 sleep、usleep 这种定时挂起与唤醒，另一方面负责 IO 操作的超时控制。比如 connect 或 read 长时间没有完成，就可以通过定时器触发超时回调，避免协程无限期挂起。

### 13. epoll 为什么适合这个项目？

因为 epoll 适合同时监听大量 fd，只在事件真正发生时通知程序，避免了遍历所有连接或频繁轮询的开销。协程库需要知道“哪个 IO 已经就绪”，而 epoll 正好提供了这种高效的事件通知机制。

### 14. 这个项目最大的亮点是什么？

我觉得最大的亮点是把协程、调度器、epoll、定时器和 hook 串成了一条完整执行链路。它不是单纯做了一个 Fiber 切换 demo，而是已经具备了协程运行时框架的基本形态，能够比较完整地展示高并发 IO 的处理思路。

### 15. 这个项目有什么不足或者可以优化的地方？

首先它基于 ucontext，作为教学和理解机制很好，但工程上相对传统；其次目前更偏底层运行时框架，缺少更完整的业务场景验证，比如高并发服务 benchmark、共享栈、work stealing 调度策略、更完善的 hook 覆盖等，后续还有比较多可扩展空间。
